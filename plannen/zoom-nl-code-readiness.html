Zoom.nl · Code Readiness Plan

Van Vibe Coding naar
Veilig Samenwerken met AI

Hoe we de zoom-nl codebase klaarmaken zodat het hele team veilig en efficient kan bouwen met Claude Code. Van guard rails tot samenwerkingsafspraken.

Methodiek

Dit plan is gebaseerd op de AI Engineering Gids -- alle principes, werkwijzen en samenwerkingsregels staan daar uitgelegd.

Bijgewerkt na codebase-analyse

Dit plan is bijgewerkt op basis van een grondige analyse van de zoom-nl codebase. Fouten in env vars, ontbrekende security issues en verkeerde prioriteiten zijn gecorrigeerd. Nieuwe fase 1 (Actieve Problemen) toegevoegd voor kritieke issues die direct aandacht nodig hebben.

START HIERContext & Doel

De rest van de monorepo (Astro apps, packages/ui, packages/api-new) is al grotendeels TypeScript en goed gestructureerd. Zoom.nl is de uitzondering:

598 JavaScript bestanden

Terwijl alle andere apps TypeScript-first zijn. Geen type safety, geen compiler die fouten vangt.

0 tests

Geen vangnet. Elke AI-wijziging is een gok -- je weet pas in productie of het werkt.

10 bestanden >300 regels

De foto detail pagina is 675 regels. Mixed concerns: data, logica, rendering in een bestand.

Actieve security issues

Gelekte reCAPTCHA secret key, onbeveiligde API route, cookies zonder secure/httpOnly flags.

DOELDe codebase zo inrichten dat een AI moeilijk dingen kapot kan maken

Dit plan brengt zoom-nl stap voor stap in lijn met de rest van de monorepo. Niet herschrijven -- verbeteren. Zodat AI-assistenten dezelfde patronen kunnen volgen die al werken in de andere apps.

FASE 1Actieve Problemen Oplossen 2-3 DAGEN

Waarom dit eerst

Deze issues zijn nu actief: een gelekte secret key, onbeveiligde API routes, runtime errors op elke pagina, en onnodige kosten per request. Dit heeft prioriteit boven constanten centraliseren of bestanden splitsen.

Security -- KRITIEK

1.1
reCAPTCHA secret key lekt naar client bundle

In src/services/captcha.js:4 wordt process.env.NEXT_PUBLIC_RECAPTCHA_SECRET_KEY gebruikt. De NEXT_PUBLIC_ prefix zorgt ervoor dat Next.js deze variabele in de client bundle stopt. Iedereen kan de secret key uit de browser halen.

"Hernoem NEXT_PUBLIC_RECAPTCHA_SECRET_KEY naar RECAPTCHA_SECRET_KEY (zonder NEXT_PUBLIC_ prefix). Verplaats de reCAPTCHA verificatie naar een API route (server-side only). Update .env.example."
Vercel env vars aanpassen na deploy
1.2
Kevel API route beveiligen

pages/api/tasks/kevel.js voert code uit zonder authenticatie. Voeg een auth check toe.

"Voeg authenticatie toe aan pages/api/tasks/kevel.js. Minimaal: check op een API key header of session token. Geen ongeauthenticeerde requests."
Security review
1.3
Cookie security flags toevoegen

In src/services/helpers.js (regels 193, 199, 209, 215) worden auth cookies gezet met httpOnly: false en zonder secure: true. XSS-aanvallen kunnen tokens stelen, en tokens kunnen over HTTP worden onderschept.

"Voeg secure: true en httpOnly: true toe aan alle setCookie calls in src/services/helpers.js. Zorg dat SameSite ook goed staat (Lax of Strict)."
1.4
Security headers toevoegen

CSP, Referrer-Policy en Permissions-Policy ontbreken in next.config.js. Dit is een standaard security baseline.

"Voeg security headers toe aan next.config.js via de headers() config: Content-Security-Policy, Referrer-Policy (strict-origin-when-cross-origin), Permissions-Policy, X-Content-Type-Options (nosniff), X-Frame-Options (DENY)."

Runtime bugs -- KAPOT

1.5
_app.js: UA code verwijderen + class component migreren

In pages/_app.js:75-77,180-192 staan ga() calls die runtime errors geven (UA is uitgezet door Google). Daarnaast is _app.js een class component (class ZoomApp extends React.Component) -- het enige class component in de app. Dit blokkeert het gebruik van hooks en modern React.

Uit haalbaarheidsanalyse

De haalbaarheidsanalyse (jan 2026) noemt class components in _app.js als een van de 5 complexiteitsfactoren. Migratie naar functional component maakt hooks mogelijk en is voorwaarde voor toekomstige modernisering.

"Verwijder alle Universal Analytics code uit pages/_app.js (ga() calls op regels 75-77 en 180-192). Migreer daarna de class component naar een functional component met hooks. Behoud alle bestaande functionaliteit."
1.6
App Router conflict oplossen

De app/ directory en pages/ directory zijn tegelijk actief. app/layout.js dupliceert functionaliteit uit pages/_app.js (metadata, analytics) en bevat dode UA code die runtime errors geeft.

Let op

De oorspronkelijke versie van dit plan stelde voor om GA/GTM IDs in app/layout.js naar env vars te verplaatsen. Maar dit bestand moet weg -- het is een conflict met de Pages Router die de app daadwerkelijk gebruikt.

"Verplaats eventuele unieke routes uit app/ naar pages/. Verwijder daarna de app/ directory. Zoom.nl gebruikt de Pages Router -- app/layout.js dupliceert _app.js en veroorzaakt inconsistenties."
Check of er routes in app/ zitten die niet in pages/ bestaan
1.7
process.browser vervangen (21 locaties)

process.browser is deprecated in Next.js en kan breken bij een upgrade. Vervang door typeof window !== 'undefined'.

"Vervang alle 21 voorkomens van process.browser in 17 bestanden door typeof window !== 'undefined'. Gebruik zoek-en-vervang, niet handmatig."
1.8
getInitialProps vervangen

pages/artikel/[[...slug]]/index.js:20-44 gebruikt getInitialProps (deprecated). Migreer naar getServerSideProps.

"Vervang getInitialProps in pages/artikel/[[...slug]]/index.js door getServerSideProps. Behoud dezelfde data-fetching logica."

Kosten -- RISICO

1.9
Middleware redirect-caching

middleware.js:57 doet een GraphQL-call bij elke request. De TTL-parameter in src/lib/apollo.js:10 wordt nooit gebruikt. Dit is het grootste Vercel-kostenrisico.

"Voeg per-URL caching toe aan de middleware redirect-lookup. Gebruik een in-memory cache met TTL (bijv. 5 minuten). De huidige code doet een GraphQL-call bij elke request zonder caching."

Configuratie opschonen

1.10
ESLint config aanmaken (incl. max-lines regel)

Er is geen .eslintrc in zoom-nl. De monorepo heeft een shared config in packages/eslint-config-custom/ die niet wordt gebruikt. Zonder ESLint config werkt de lint-hook uit stap 0.5 niet.

Uit refactoring plan

Het refactoring plan beveelt een max-lines ESLint regel aan die de 200 LOC limiet afdwingt. Dit voorkomt dat nieuwe bestanden te groot worden na fase 3 (splits).

"Maak .eslintrc.js aan die extends van eslint-config-custom/next.js. Voeg een lint script toe aan package.json. Voeg de regel max-lines toe: ['warn', { max: 200, skipBlankLines: true, skipComments: true }]."
1.11
Tailwind content-paths fixen

tailwind.config.js mist ./src/**/* in de content-array. Tailwind classes in src/ worden niet opgepikt.

"Voeg './src/**/*.{js,ts,jsx,tsx}' toe aan de content-array in tailwind.config.js."
1.12
tsconfig.json extend shared config

De huidige tsconfig.json definieert alles zelf, terwijl de monorepo een shared tsconfig/nextjs.json heeft. allowJs: true en strict: true staan al aan -- die hoeven niet meer gezet te worden.

Correctie

De oorspronkelijke versie van dit plan (fase 5.1) stelde voor om allowJs: true aan te zetten. Dit staat al aan in tsconfig.json:14. De actie is: extend de shared config en verwijder duplicate opties.

"Pas tsconfig.json aan: extend tsconfig/nextjs.json (de shared monorepo config). Verwijder opties die al in de shared config staan. Behoud alleen zoom-nl-specifieke overrides."
Verificatie

Security: grep -r "NEXT_PUBLIC_RECAPTCHA_SECRET" src/ geeft 0 resultaten. Cookies hebben secure + httpOnly. Security headers in response.
Runtime: Geen ga() calls, geen app/ directory, process.browser grep geeft 0 resultaten.
Config: npx next lint werkt, Tailwind classes in src/ worden opgepikt, tsconfig extend shared.

Livegang

Hoog risico bij security fixes -- test auth flows, reCAPTCHA, en cookie-gebaseerde sessies grondig op staging. Middleware-caching apart deployen en monitoren op Vercel. Runtime fixes (UA code, process.browser) zijn laag risico.

FASE 3Grote Bestanden Opsplitsen 3-5 DAGEN

Waarom dit de grootste verbetering is

Bij een 675-regel bestand moet een AI het hele bestand lezen om iets kleins te wijzigen. Hogere kans op fouten, meer tokens, hogere kosten. Vuistregel: max 150 regels per bestand.

Belangrijk

Gebruik /plan voor elke split. Laat Claude eerst analyseren welke state, hooks en functies er zijn, en hoe ze opgesplitst kunnen worden. Akkoord geven, dan pas uitvoeren.

Strangler fig pattern (uit refactoring plan)

Per bestand: (a) analyseer dependencies, (b) schrijf tests voor bestaand gedrag eerst, (c) extract naar nieuwe modules met tests, (d) verwijder oude code. Tests VOOR de split schrijven is veiliger -- je vangt regressies direct op.

Git-discipline bij splits

Elke split = eigen branch + eigen PR (max ~300 regels). Commit na elke geslaagde stap met specifieke bestanden (git add <bestand>, nooit git add .).

Buiten scope: toekomstige consolidatie

Styling consolidatie (SCSS -> Tailwind)

Zoom.nl gebruikt drie styling-systemen door elkaar: SCSS modules, Tailwind CSS en styled-components (v6.1.0). De rest van de monorepo gebruikt Tailwind. Deze mix is verwarrend voor AI-assistenten en maakt consistente styling moeilijk.

Volledige SCSS -> Tailwind migratie is een apart project dat buiten dit plan valt. Aanpak voor nu: nieuwe code schrijft Tailwind. Bij het aanraken van een bestaand component (bugfix, feature, split) migreer je de styling mee. Geen big bang -- geleidelijk, per component.

Stap 2.5 in dit plan inventariseert welke componenten welk systeem gebruiken, zodat AI's weten welk patroon ze moeten volgen.
