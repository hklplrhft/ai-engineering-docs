DEEL 14Parallel Werken

Meerdere Claude Code-instanties tegelijk, elk op een eigen branch/worktree.

Worktrees -- eenvoudig parallel werken

# Start Claude in een eigen worktree (eigen branch + bestanden)
claude --worktree feature-auth

# Start nog een sessie in een aparte worktree
claude --worktree bugfix-123

# Of laat Claude zelf een worktree maken
"Werk in een worktree voor deze feature"

Worktrees worden automatisch opgeruimd als er geen wijzigingen zijn.

Writer/Reviewer patroon

Een frisse context reviewt beter -- Claude is niet bevooroordeeld naar code die het net zelf schreef.

Sessie A (Writer)"Implementeer een rate limiter voor onze API endpoints"Schrijft de code
Sessie B (Reviewer)"Review de rate limiter in @src/middleware/rateLimiter.ts op edge cases en race conditions"Reviewt met frisse blik
Sessie A (Writer)"Hier is de review feedback: [output]. Los deze issues op."Verwerkt feedback

Aandachtspunten: Aparte branches (voorkom merge-conflicten). Elk op ander deel van codebase. Controleer plan-limieten.

Samenwerken in een team -- conflicten voorkomen

De gouden regel

Werk nooit tegelijk aan hetzelfde bestand. AI herschrijft grote blokken code -- merge-conflicten zijn moeilijk te resolven en kosten meer tijd dan ze besparen.

Bestandsverdeling strategie

Verdeel op bestandsniveau, niet op feature-niveau. Plan vooraf wie welke bestanden raakt. Twee developers aan dezelfde feature is prima, zolang ze verschillende bestanden aanpassen.

develop
  |-- feature/hero-section          (Developer A: src/components/hero/)
  |-- feature/dark-mode             (Developer B: src/styles/, src/hooks/useTheme.js)
  |-- refactor/split-photo-detail   (Developer C: pages/foto/, src/hooks/usePhoto*)

Coordinatie

Check voor je begint of iemand recent aan hetzelfde bestand heeft gewerkt:

git log --all --oneline --since="3 days ago" -- <bestand>

Splits grote features in stappen die verschillende bestanden raken: eerst nieuwe bestanden (geen conflict), dan kleine wijzigingen aan bestaande (snel mergen).

PR-regel

Nooit twee open PRs die hetzelfde bestand raken. Bij overlap: een developer gaat eerst en merget. De tweede rebaset op develop.
