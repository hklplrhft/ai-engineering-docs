DEEL 3Het Instructiesysteem — De 5 Lagen

Het hart van AI-engineered development. Bouw op in deze volgorde.

Waar hoort wat? -- Snelgids

~/.claude/CLAUDE.mdPersoonlijke regels voor al je projecten. Werkwijze, stijl, veiligheid."Werk altijd op een branch" · "Comments in het Nederlands" · werkwijze-signalen
./CLAUDE.mdProjectspecifieke briefing. Tech stack, commando's, kritieke regels, verwijzingen."Gebruik Prisma migrate, nooit raw SQL" · "Zie docs/refs/db-schema.md"
docs/refs/Gedetailleerde "littekens" -- lessen uit fouten. Worden on-demand geladen."Kolom is is_answer_ready, NIET answer_ready" · API-valkuilen · deploy-stappen
MEMORY.md + auto memoryPersistent geheugen per project, overleeft sessies. Niet in je repo -- staat in ~/.claude/projects/. Persoonlijk: elke ontwikkelaar bouwt eigen geheugen op, ook in dezelfde repo."Bug tracker staat in Linear project INGEST" · "Gebruiker prefereert korte antwoorden"
Vuistregel

Geldt voor elk project? → globale CLAUDE.md. Specifiek voor dit project? → project CLAUDE.md. Gedetailleerde les uit een fout? → docs/refs/. Context die niet in code of git staat? → memory. Dupliceer niet tussen lagen -- verwijs.

LAAG 1Globale regels~/.claude/CLAUDE.md

Persoonlijke voorkeuren voor alle projecten. "Schrijf comments in het Nederlands." · "Gebruik hyphens, nooit em-dashes." · "Vraag bevestiging voor verwijderen."

Vuistregel: Geldt voor elk project → hier. Projectspecifiek → niet.

De globale CLAUDE.md als poortwachter

Laag 1 is meer dan stijlvoorkeuren -- het is het mechanisme om je hele werkwijze af te dwingen. Door twee compacte secties toe te voegen aan je globale CLAUDE.md, transformeer je het van passief naar actief:

Twee toevoegingen die alles veranderen

1. Project-inrichting checks -- Claude controleert bij elke sessie of de basis aanwezig is: CLAUDE.md, hooks, docs/refs/ en .gitignore. Ontbreekt er iets? Claude biedt aan om het op te zetten.

2. Werkwijze-signalen -- Claude signaleert automatisch op cruciale momenten: bugfix → regel aanbieden, herhaling → skill voorstellen, terugkerende controle → agent aanbieden, vastlopen → reset suggereren, taakwissel → /clear herinneren, einde → handoff aanbieden, bloat → snoei waarschuwen, nieuwe code → test herinneren.

Dit implementeert de feedback loop (Laag 3) en de anti-patronen automatisch. De concrete copy-paste configuratie vind je in de Starter Kit (stap 2).

LAAG 2Projectregels./CLAUDE.md

Briefing voor je specifieke project. Automatisch geladen. Houd het compact -- in de praktijk blijkt 400+ regels te veel; ~100 regels werkt beter. Frontier modellen volgen ~150-200 instructies, minus ~50 voor Claude Code's systeemprompt. Je hebt ~100-150 regels ruimte.

Erin: Project overview, tech stack, kritieke regels, verwijzingen naar ref-docs, testcommando's.
Niet erin: Code style (→ linter/hook), code-voorbeelden (→ verouderen), afleidbare informatie.

Progressive disclosure -- kleine bestanden

CLAUDE.md is een inhoudsopgave, niet het hele boek. Stop details in aparte bestanden en verwijs ernaar. Claude laadt ze wanneer nodig.

Voorbeeld bestandsstructuur

mijn-project/
├── CLAUDE.md              ← kort: overzicht + verwijzingen
├── docs/
│   └── refs/
│       ├── api-regels.md  ← API conventies en valkuilen
│       ├── db-schema.md   ← kolomnamen, relaties, gotchas
│       └── deploy.md      ← deployment stappen
├── .claude/
│   ├── settings.json      ← permissions, hooks
│   ├── agents/            ← project agents
│   └── skills/            ← project skills
└── src/                   ← je code

Voorbeeld CLAUDE.md

# Mijn Project

## Overview
Webshop gebouwd met Next.js 14, Prisma ORM, PostgreSQL.

## Tech stack
- Frontend: React + Tailwind CSS
- Backend: Next.js API routes
- Database: PostgreSQL via Prisma
- Tests: Jest + React Testing Library

## Kritieke regels
- ALTIJD Prisma migrate gebruiken, nooit raw SQL voor schema changes
- Componenten in src/components/, pages in src/app/
- Zie docs/refs/api-regels.md voor API conventies
- Zie docs/refs/db-schema.md voor database valkuilen

## Commando's
- Test: npm test
- Lint: npm run lint
- Build: npm run build
- Dev: npm run dev

Van gesprek naar regel

Gespreksinstructies zijn sessie-only

Zeg je in een gesprek: "Gebruik nooit de externe Geocoding API" -- dan geldt dat alleen in die sessie. Volgende sessie is het vergeten. Wil je het waarborgen? Zet het in je project CLAUDE.md.

Voorbeeld: Je ontdekt dat de externe Geocoding API onbetrouwbaar is en zegt tegen Claude: "Gebruik altijd onze interne location service, nooit de Geocoding API." Zo maak je het permanent:

# ./CLAUDE.md in je project-root (voeg toe aan kritieke regels)
- Gebruik ALTIJD de interne location service (/api/location)
  Nooit de externe Geocoding API -- die is onbetrouwbaar en kost per request

Na deze toevoeging geldt de regel in elke toekomstige sessie, voor iedereen die het project cloned.

Tip: laat Claude het onthouden

Voeg een meta-regel toe aan je CLAUDE.md zodat Claude zelf aanbiedt om gespreksregels permanent te maken:

# ./CLAUDE.md
## Werkwijze
- Wanneer ik in een gesprek een regel geef die voor het hele project moet gelden,
  bied dan aan om deze toe te voegen aan ./CLAUDE.md (of docs/refs/ als het uitgebreid is).

Quick start: draai /init om een starter CLAUDE.md te genereren.

LAAG 3Taakspecifieke regelsdocs/refs/

Je "littekens" -- lessen van productiefouten. Laden op aanvraag. Claude maakt fout → je schrijft regel → nooit meer herhaald.

# docs/refs/db-schema.md

## Valkuilen
- De kolom is `is_answer_ready`, NIET `answer_ready`
- Gebruik altijd soft deletes (deleted_at), nooit hard deletes
- Foreign keys MOETEN ON DELETE CASCADE hebben

## Naamconventies
- Tabellen: meervoud, snake_case (users, order_items)
- Kolommen: snake_case (created_at, is_active)
Hoe het werkt

CLAUDE.md zegt: "Zie docs/refs/db-schema.md voor database valkuilen." Claude leest dat bestand alleen wanneer hij met de database werkt. Zo blijft je hoofdbestand klein en je context schoon.

De Feedback Loop

Fout gevonden
→
Regel toegevoegd
→
Nooit meer herhaald
↩

Elke fout wordt een regel. Na een maand: tientallen regels. Na drie maanden: het systeem kent jouw project beter dan welke nieuwe developer ook.

Concreet voorbeeld: van fout naar regel

Situatie: Claude verwijdert een bestand uit public/js/. Blijkt: dat bestand wordt automatisch gegenereerd door een build-stap. Productie breekt.

StapActieVoorbeeld
1. FixLos het probleem opHerstel het bestand, draai de build opnieuw
2. TestSchrijf een regressietestTest dat public/js/ compleet is na build
3. RegelVoeg toe aan docs/refs/Verwijder NOOIT bestanden uit public/js/
4. CLAUDE.mdVerwijs naar de regelZie docs/refs/build-regels.md
5. HookBlokkeer bij kritieke foutennpm run build && git diff --exit-code public/js/
In de praktijk

De /fix-bug skill (Starter Kit, stap 6) automatiseert dit patroon. Eén commando doorloopt alle stappen -- van branch aanmaken tot regressietest, regel in docs/refs/, en commit.

Het patroon

Fout → fix → test → regel → (hook). Hoe kritischer de fout, hoe harder de garantie. Vervelend? Regel. Productie kapot? Hook. Dit is hoe je systeem steeds slimmer wordt.

LAAG 4Persistent geheugenMEMORY.md + auto memory

Overleeft sessies. Bekende bugs, beslissingen, project IDs, procedures. Zonder dit begint elke sessie opnieuw. Memory is per project -- opgeslagen in ~/.claude/projects/, niet in je repo.

Handmatig geheugen

/memory om de project-specifieke memory bestanden te bewerken. Schrijf regels, beslissingen en context op die je wilt behouden.

Auto memory

Claude leert automatisch tijdens het werken. Build-commando's, debugging-inzichten, project-patronen -- worden opgeslagen zonder dat je iets hoeft te schrijven. Schakel in/uit via /memory.

Twee vormen, samen sterker

Auto memory vangt operationele kennis op (hoe te builden, welke tests draaien). Handmatig geheugen voor architectuurbeslissingen en regels die Claude moet volgen. Gebruik beide.

Handoff files

Beëindig je een sessie? Vraag Claude om HANDOFF.md te schrijven in de root van je project. Dit is geen log -- het is een briefing voor de volgende sessie.

Inhoud van een HANDOFF.md

Doel: Wat probeerde je te bereiken?
Status: Waar sta je nu? Wat werkt, wat niet?
Geprobeerd: Welke aanpak heb je gevolgd? Wat faalde en waarom?
Volgende stappen: Wat moet er nog gebeuren? In welke volgorde?
Open vragen: Waar liep je vast? Wat moet iemand anders beslissen?

Vervangen, niet stapelen. Er is altijd maar één HANDOFF.md. Elke sessie overschrijft het vorige bestand met de actuele stand van zaken. Het is geen changelog -- het is de huidige status. Oude context zit in git history en in memory.

Wanneer schrijven? Aan het einde van een sessie waarin je iets complex hebt gedaan dat niet af is. Niet nodig na een simpele bugfix die al gecommit is. Wel nodig als een collega (of jijzelf morgen) anders niet weet waar je gebleven bent.

# HANDOFF.md
## Doel
Upload dialog opsplitsen in kleinere componenten (fase 3.2)

## Status
Upload-form component is klaar en getest. Preview-stap nog niet begonnen.

## Geprobeerd
- useUpload hook geëxtraheerd -- werkt
- Upload-form afgesplitst -- werkt
- Preview component begonnen maar state-sync tussen form en preview is lastiger dan verwacht

## Volgende stappen
1. State lifting: upload state naar dialog-upload.js tillen
2. Preview component afsplitsen
3. Handmatig testen van hele upload flow

## Open vragen
- Moet de preview stap een eigen API call doen of hergebruiken we de form data?
LAAG 5Uitvoerbare extensiesskills, agents, hooks

Code die Claude niet alleen leest maar uitvoert. Van "de AI helpt mij" naar "het systeem werkt voor mij." Vier bouwstenen:

SkillsHerbruikbare instructieset (recept). Claude volgt de stappen in je huidige sessie.Taak 3x handmatig gedaan? Maak een skill.
AgentsZelfstandige Claude-instantie met eigen context. Draait parallel, eigen focus.Werk isoleren, parallel reviewen, bulk taken.
HooksShell-script dat automatisch draait. Deterministisch -- kan niet genegeerd worden.Alles wat kapot gaat als Claude het negeert.
TestsBewijs dat code werkt. Vangt regressies op voor ze in productie komen.Elke functie die fout mag gaan = een test.

Kernonderscheid: Skills en agents zijn probabilistisch -- Claude kan afwijken. Hooks en tests zijn deterministisch -- ze blokkeren altijd. Gebruik beide lagen samen.
