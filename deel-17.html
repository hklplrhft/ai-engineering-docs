DEEL 17De Feedback Loop

Fout gevonden
→
Regel toegevoegd
→
Nooit meer herhaald
↩

Elke fout wordt een regel. Na een maand: tientallen regels. Na drie maanden: het systeem kent jouw project beter dan welke nieuwe developer ook.

Concreet voorbeeld: van fout naar regel

Situatie: Claude verwijdert een bestand uit public/js/. Blijkt: dat bestand wordt automatisch gegenereerd door een build-stap. Productie breekt.

Stap 1: Fix de fout

"Herstel het verwijderde bestand en draai de build opnieuw"

Stap 2: Schrijf een regressietest

"Schrijf een test die verifieert dat alle verwachte
 bestanden in public/js/ aanwezig zijn na een build"

Stap 3: Voeg een regel toe

# docs/refs/build-regels.md
- Verwijder NOOIT bestanden uit public/js/ --
  deze worden automatisch gegenereerd door de build.
  Check eerst: npm run build

Stap 4: Verwijs vanuit CLAUDE.md

# CLAUDE.md (voeg toe aan kritieke regels)
- Zie docs/refs/build-regels.md voor build-gerelateerde valkuilen

Stap 5 (optioneel): Maak een hook

Als de fout echt kritiek is, blokkeer het met een hook zodat het onmogelijk wordt:

# Pre-commit hook die checkt of gegenereerde bestanden intact zijn
npm run build && git diff --exit-code public/js/
Het patroon

Fout → fix → test → regel → (hook). Hoe kritischer de fout, hoe harder de garantie. Vervelend? Regel. Productie kapot? Hook. Dit is hoe je systeem steeds slimmer wordt.
