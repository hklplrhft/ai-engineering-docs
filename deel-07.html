DEEL 7Tests -- Het Bewijs DETERMINISTISCH

Tests bewijzen dat je code werkt -- en blijft werken na wijzigingen. Zonder tests is elke verandering een gok.

Drie soorten tests

Unit testsTesten individuele functies -- snel, vangen logica-foutenMeest voorkomend
Integration testsTesten componenten samen -- API + database, frontend + backendTrager, maar realistischer
Golden file testsVergelijken output met vastgelegde correcte versieGoed voor HTML/JSON output

Voorbeeld: laat Claude tests schrijven

# Vraag Claude om tests te schrijven
"Schrijf tests voor de calculateDiscount functie
 die alle edge cases dekt"

# Claude genereert bijv.:
describe('calculateDiscount', () => {
  test('applies 10% for orders over 100', () => {
    expect(calculateDiscount(150)).toBe(135);
  });

  test('no discount for orders under 100', () => {
    expect(calculateDiscount(50)).toBe(50);
  });

  test('handles zero amount', () => {
    expect(calculateDiscount(0)).toBe(0);
  });

  test('handles negative amount', () => {
    expect(() => calculateDiscount(-10)).toThrow();
  });
});

De automatische feedback loop

Claude schrijft code
→
Hook draait tests
→
Test faalt
→
Claude ziet fout
→
Claude fixt
→
Tests slagen
→
Commit

Dit hele proces verloopt zonder jouw tussenkomst. De post-write hook (zie Hooks) draait tests na elke wijziging. Claude leest de foutmelding en past de code aan tot alles slaagt.

Zo begin je

  1. Vraag Claude: "Welk test framework past bij dit project?"
  2. "Stel het test framework in en maak een eerste test"
  3. "Schrijf tests voor [functie X] die alle edge cases dekt"
  4. Stel een post-write hook in die tests automatisch draait
  5. Bij elke bugfix: "Fix de bug en schrijf een regressietest die voorkomt dat dit terugkomt"
Gouden regel

Elke bugfix verdient een test. Zonder test is een fix tijdelijk -- dezelfde bug kan morgen terugkomen. Claude is verrassend goed in edge cases die je zelf mist.
