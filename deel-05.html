DEEL 5Agents — De Specialisten PROBABILISTISCH

Zelfstandige Claude Code-instantie met eigen context. Skill = recept in je huidige sessie. Agent = aparte kok die je oproept.

Waarom: Focus (weet alleen wat relevant is), schone context (start fris), parallellisme (meerdere tegelijk), kosten besparen (goedkoper model per agent).

Ingebouwde agents

ExploreRead-only, codebase doorzoeken (Haiku -- snel & goedkoop)Kan niets wijzigen
PlanResearch voor Plan Mode (erft model)Read-only
General-purposeComplexe multi-step taken (erft model, alle tools)Breed inzetbaar

Eigen agents maken

Markdown-bestand met YAML frontmatter. Maak via /agents of handmatig.

---
name: code-reviewer
description: Reviews code for quality and best practices.
  Use proactively after code changes.
tools: Read, Grep, Glob, Bash
model: sonnet
---

You are a senior code reviewer. When invoked:
1. Run git diff to see recent changes
2. Review for quality, security, maintainability
3. Provide feedback: critical / warnings / suggestions

Agent memory -- persistent leren

Met memory: user onthoudt een agent wat hij leert over sessies heen. Na 10x code reviewen kent hij je patronen, conventies en terugkerende problemen.

---
name: code-reviewer
description: Reviews code for quality
memory: user       # leert over alle projecten
# memory: project  # leert alleen binnen dit project
# memory: local    # project-specifiek, niet in git
---

Workflows

Foreground vs background

Foreground: blokkeert je sessie, kan vragen stellen. Background: draait concurrent, auto-deniet ontbrekende permissions. Druk Ctrl+B om een lopende agent naar de achtergrond te sturen.

Resume -- eerder werk voortzetten

Elke agent-aanroep start normaal fris. Maar je kunt vragen om voort te bouwen op eerder werk:

"Gebruik de code-reviewer om de auth module te reviewen"
[Agent klaar]
"Ga verder met die review en bekijk nu de authorization logica"
→ Claude hervat de agent met volledige context

Chaining -- agents in serie

Resultaat van agent A → input voor agent B:

"Gebruik code-reviewer om performance issues te vinden,
 gebruik daarna optimizer om ze te fixen"

Parallel research

"Onderzoek de auth, database en API modules parallel
 met aparte agents"

Wanneer wat gebruiken?

Main conversationIteratief werk, snelle wijzigingen, gedeelde contextContext vult sneller
SubagentVerbose output isoleren, tool-restricties, zelfstandig werkLatency bij opstarten
SkillHerbruikbare prompts die in main context draaienDeelt context
/btwSnelle side question, geen tools, niet in historieGeen tool access
Agent TeamMeerdere agents die parallel coordineren, eigen contextComplexer setup

Patronen

Bouwers: content-writer, explore. Bewakers: audit-seo, audit-db, security-advisor.

Orchestrator-patroon: Een meta-agent (audit-all) stuurt specialisten aan.

Best practices

1. Elke agent moet uitblinken in een specifieke taak -- niet alles-in-een.
2. Schrijf gedetailleerde descriptions -- Claude gebruikt dit om te delegeren.
3. Beperk tool access tot wat nodig is (security + focus).
4. Check project agents in version control -- deel met je team.
5. Gebruik "proactively" in description als de agent automatisch moet activeren.
