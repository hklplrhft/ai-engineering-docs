DEEL 22Starter Kit -- Direct Goed Beginnen

Je hoeft niet alles uit deze gids te onthouden. Met de juiste inrichting dwingt Claude Code de werkwijze grotendeels automatisch af. Volg de stappen hieronder om je eigen setup op te bouwen, en iedereen in je team werkt meteen volgens best practices.

Wat wordt afgedwongen vs. wat vereist discipline?

Automatisch (hooks): lint, typecheck, tests, secret detection, project-check bij sessiestart -- Claude kan niet committen zonder, en vergeet de basis niet.
Sterk gestuurd (CLAUDE.md + skills): branches, plan mode, commit-messages, werkwijze.
Vereist discipline: code reviewen, goede prompts schrijven, /clear gebruiken.

SSOT-principe

Elke regel heeft een autoritieve bron. Globale werkwijze → ~/.claude/CLAUDE.md. Projectregels → ./CLAUDE.md. Valkuilen → docs/refs/. Andere lagen verwijzen naar de bron, kopieren niet. Bij conflict wint de bron. Zie De 5 Lagen -- SSOT voor details.

Stap 1: Installeer Claude Code

# macOS / Linux
npm install -g @anthropic-ai/claude-code

# Of via installer script (macOS / Linux)
# curl -fsSL https://claude.ai/install.sh | bash

# Start Claude in je project
cd mijn-project && claude

Stap 2: Maak de globale CLAUDE.md

Maak ~/.claude/CLAUDE.md aan. Dit geldt voor al je projecten.

# ~/.claude/CLAUDE.md -- Globale regels

## Taal
- Communiceer in het Nederlands
- Code, commits en comments in het Engels

## Werkwijze -- plan, verifieer, commit

### Altijd plannen voor doen
- /plan bij taken die meer dan 1 bestand raken of langer dan 5 minuten duren
- Geef verificatiecriteria mee: welke tests moeten slagen, welke output wordt verwacht
- Wacht op goedkeuring voordat je begint met implementeren

### Context schoon houden
- Stel /clear voor bij taakwissel (andere feature, ander onderwerp)
- Stel /compact voor na 30+ minuten of wanneer context vol raakt
- Bij sessie-overdracht: maak een HANDOFF.md (doel, status, wat geprobeerd, volgende stappen)

### Git-discipline
- Werk op branches, niet direct op main
- Commit na elke voltooide wijziging -- herinner als er nog niet gecommit is
- Selectief committen: specifieke bestanden, niet git add .
- Nooit pushen, deployen of berichten sturen zonder expliciete bevestiging

### Verificatie
- Waarschuw bij hallucinaties (onbekende packages, API's, functies) -- verifieer eerst
- Stel git diff voor bij grotere wijzigingen zodat de gebruiker kan reviewen
- Commit nooit code die de gebruiker niet begrijpt -- bied uitleg aan

## Code kwaliteit

### Test-first bij bugs
- Bij bugfixes: schrijf eerst een falende regressietest, dan pas de fix
- Test na elke wijziging -- herinner aan testen/lint/analyze na code changes
- Zonder test is een fix alleen voor vandaag

### Feedback loop -- van fout naar regel
- Fout gevonden -> fix + regressietest
- Terugkerend probleem -> regel toevoegen aan docs/refs/ of CLAUDE.md
- Kritieke fout -> stel een hook voor die het afdwingt
- Doel: elke fout kan maar een keer voorkomen

### Kwaliteitscontroles
- Geen secrets hardcoden -- gebruik environment variables of .env + .gitignore
- Review gegenereerde code op OWASP top 10 (SQL injection, XSS, command injection)
- Stel linters/typecheck voor als die ontbreken in het project

## Project-inrichting check
Bij het begin van een taak in een nieuw project, doorloop de volledige setup:
- Lees ~/.claude/docs/project-setup.md en doorloop alle stappen
- Stel elke stap voor aan de gebruiker voordat je begint met de eigenlijke taak

Minimale check bij bestaande projecten:
- CLAUDE.md van het actieve project (zo niet: stel /init voor)
- Hooks voor lint/typecheck (zo niet: stel voor om in te richten)
- docs/refs/ map voor littekens-documenten (zo niet: vermeld het bij terugkerende fouten)
- .gitignore met .env en secrets (zo niet: waarschuw)

## Signalen -- wanneer ingrijpen
- Na 2 mislukte pogingen voor dezelfde taak: stel voor om te stoppen, /clear, en een
  betere prompt te formuleren
- Bij herhaling van eerder gemaakte fouten: stel voor om een regel/hook toe te voegen
- Bij vage opdrachten: vraag om specifieke bestanden, error output, of verwacht gedrag
- Bij grote wijzigingen zonder tests: weiger door te gaan totdat er verificatiecriteria zijn

## Signalen -- wanneer uitbreiden (5-lagen systeem)
Stel de volgende uitbreidingen voor wanneer het moment er is. Niet vooraf, maar zodra
de situatie zich voordoet:
- Bug gefixt zonder test? -> "Zal ik een test-framework opzetten?"
- Zelfde fout 2x? -> "Zal ik een hook instellen?"
- Valkuil ontdekt? -> "Zal ik dit in docs/refs/ vastleggen?"
- Handeling 3x herhaald? -> "Zal ik hier een skill van maken?"
- Parallelle controle nodig? -> "Zal ik een custom agent maken?"
- Sessie-einde met onafgemaakt werk? -> "Zal ik een HANDOFF.md maken?"

## Veiligheid
- Review shell commando's kritisch, vooral: rm -rf, git push --force, drop table
- Geen gevoelige data (API keys, wachtwoorden) in prompts of CLAUDE.md
- Stel nooit voor om hooks te skippen (--no-verify) of checks uit te schakelen

## Stijl
- Geen em-dashes of en-dashes in code of tekst, gebruik --
- Wees beknopt in uitleg, gedetailleerd in code
Hoe werken deze secties samen?

Project-inrichting check verwijst naar ~/.claude/docs/project-setup.md voor de volledige setup-procedure (SSOT -- details staan op een plek). De minimale check zorgt dat bestaande projecten ook gecontroleerd worden.

Signalen "wanneer ingrijpen" voorkomen de veelgemaakte fouten: eindeloos herhalen, vage prompts, code zonder tests.

Signalen "wanneer uitbreiden" implementeren de 5-lagen feedback loop: elke fout wordt een regel, elke herhaling een skill, en je project groeit organisch mee.

Combineer dit met een SessionStart hook (stap 4b) voor deterministische controle van de basis.

# In je project directory

/init

# Verfijn het resultaat -- voeg toe:
# - Verwijzingen naar docs/refs/ voor specifieke regels

Globaal vs. project -- waar hoort wat?

Regels die voor elk project gelden (werkwijze, veiligheid, stijl) horen in je globale ~/.claude/CLAUDE.md (stap 2). In je project CLAUDE.md zet je alleen wat specifiek is voor dit project: commando's, conventies, architectuurkeuzes. Dupliceer geen globale regels -- Claude leest beide bestanden automatisch.

Hier zijn kant-en-klare voorbeeldzinnen voor je project CLAUDE.md. Alleen project-specifieke zaken:

## Commando's
- Tests draaien: `npm test` / `pytest` / `flutter test`
- Linting: `npm run lint` / `ruff check .`
- Build: `npm run build` / `flutter build`

## Projectregels
- Volg de conventies in docs/refs/ voor architectuur- en codebeslissingen.
- Gebruik bestaande utilities en patterns voordat je nieuwe maakt.
- Voeg geen dependencies toe zonder overleg.

En als je globale ~/.claude/CLAUDE.md uit stap 2 nog niet compleet is, vul aan met:

## Werkwijze
- Werk altijd op een feature branch, nooit direct op main of develop.
- Lees bestaande code voordat je wijzigingen voorstelt -- begrijp de context.
- Stel een regressietest voor bij elke bugfix.
- Commit regelmatig met duidelijke, Engelstalige commit messages.
- Push of deploy nooit zonder expliciete bevestiging.

## Veiligheid
- Geen secrets hardcoden -- gebruik environment variables of .env (met .gitignore).
- Review gegenereerde code op OWASP top 10 kwetsbaarheden.
- Wees extra voorzichtig met destructieve shell commando's (rm -rf, git push --force, DROP TABLE).

## Stijl
- Code, commits en comments in het Engels.
- Wees beknopt in uitleg, gedetailleerd in code.
- Geen emoji's tenzij expliciet gevraagd.
Tip

Oefen ook meteen met /plan (denken voor doen), /clear (context opschonen bij taakwissel) en /compact (lange sessies inkorten). Dit zijn de drie commando's die je het vaakst zult gebruiken.

Stap 4: Stel hooks in

Dit is het belangrijkste deel -- hooks zijn onmogelijk te omzeilen. Maak .claude/settings.json aan in je project:

{
  "hooks": {
    "PreCommit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint && npm run typecheck"
          },
          {
            "type": "command",
            "command": "grep -rn 'sk_live\\|AKIA\\|ghp_\\|-----BEGIN' --include='*.js' --include='*.ts' --include='*.py' --include='*.env' . && echo 'BLOCKED: possible secret detected' && exit 1 || true"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm test -- --related 2>&1 | head -50"
          }
        ]
      }
    ],
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude is klaar\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}
Wat doet dit?

PreCommit: Elke commit wordt geblokkeerd als lint of typecheck faalt, of als er een secret-patroon wordt gevonden (zoals sk_live, AKIA, of ghp_). Onmogelijk te omzeilen.
PostToolUse: Na elke bestandswijziging draaien automatisch de gerelateerde tests. Claude ziet fouten en fixt ze zelf.
Notification: Desktop-melding als Claude klaar is of op je wacht.

Dit voorbeeld gebruikt Node.js-commando's. Werk je met Flutter, Python of Go? Zie Hooks -- Voorbeelden per stack voor de juiste commando's.

Stap 4b: SessionStart hook -- deterministische project-check

De signalen in je globale CLAUDE.md zijn probabilistisch -- Claude kan ze vergeten. Voeg een SessionStart hook toe die bij elke sessie deterministisch controleert of de basis aanwezig is.

Maak ~/.claude/hooks/project-check.sh aan:

#!/bin/bash
# Checkt of de basis project-inrichting aanwezig is
warnings=""

[ ! -d ".git" ] && warnings="${warnings}\n- Geen git repo. Overweeg: git init"
[ ! -f "CLAUDE.md" ] && warnings="${warnings}\n- Geen CLAUDE.md. Stel voor: /init"
[ ! -d "docs/refs" ] && warnings="${warnings}\n- Geen docs/refs/ map"
if [ -f ".gitignore" ]; then
  grep -q "^\.env" .gitignore 2>/dev/null || \
    warnings="${warnings}\n- .env staat niet in .gitignore!"
else
  warnings="${warnings}\n- Geen .gitignore gevonden"
fi

if [ -n "$warnings" ]; then
  echo "PROJECT-CHECK: Ontbrekende onderdelen:"
  echo -e "$warnings"
  echo ""
  echo "Bied aan om ontbrekende onderdelen op te zetten."
fi

Voeg toe aan je globale ~/.claude/settings.json:

{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash ~/.claude/hooks/project-check.sh"
          }
        ]
      }
    ]
  }
}
Probabilistisch + deterministisch

De SessionStart hook garandeert het fundament (CLAUDE.md, .git, .gitignore, docs/refs/). De signalen in je globale CLAUDE.md laten de rest organisch groeien (tests, skills, agents, hooks). Samen vormen ze een systeem dat niet vergeet en niet overwelmt. Zie Hooks -- SessionStart voor de uitgebreide versie.

Stap 5: Maak een start-feature skill

Maak .claude/skills/start-feature/SKILL.md aan. Dit dwingt het juiste proces af bij elke nieuwe feature.

Vuistregel: heb je een taak al 3x handmatig gedaan? Maak er een skill van.

---
name: start-feature
description: Start een nieuwe feature volgens de teamwerkwijze.
  Activeer wanneer iemand een feature, component, of pagina wil bouwen.
disable-model-invocation: true
---
Start een nieuwe feature: $ARGUMENTS

## Verplichte stappen
1. Maak een nieuwe branch: feature/[korte-naam]
2. Ga in Plan Mode en analyseer wat nodig is
3. Presenteer het plan en wacht op goedkeuring
4. Na goedkeuring: implementeer stap voor stap
5. Schrijf tests voor elke nieuwe functie
6. Draai alle tests en fix fouten
7. Commit met een beschrijvende message
8. Open een draft PR met samenvatting

## Regels
- Wijzig NOOIT bestanden buiten de scope van de feature
- Bij twijfel: vraag, implementeer niet
- Elke publieke functie krijgt minimaal 1 test

Gebruik: /start-feature login pagina met e-mail en wachtwoord

Stap 6: Maak een fix-bug skill

Maak .claude/skills/fix-bug/SKILL.md aan:

---
name: fix-bug
description: Fix een bug volgens de teamwerkwijze.
  Activeer bij bugs, errors, of onverwacht gedrag.
disable-model-invocation: true
---
Fix de volgende bug: $ARGUMENTS

## Verplichte stappen
1. Maak een branch: fix/[korte-naam]
2. Reproduceer de bug (draai de code, bekijk de error)
3. Analyseer de oorzaak (niet het symptoom)
4. Schrijf EERST een test die de bug reproduceert (test moet falen)
5. Fix de bug (test moet slagen)
6. Draai alle tests om te checken dat niets anders breekt
7. Voeg een regel toe aan docs/refs/ als dit een terugkerende fout is
8. Als de regel projectbreed geldt: verwijs ernaar vanuit CLAUDE.md
9. Bij kritieke fouten: stel een hook voor die herhaling blokkeert
10. Commit en open een draft PR

## Regels
- Fix de root cause, niet het symptoom
- Elke bugfix MOET een regressietest hebben
- Bij onduidelijke oorzaak: gebruik /plan eerst
- Stap 8-9 sluiten de feedback loop (zie deel 3: CLAUDE.md)

Gebruik: /fix-bug users krijgen een 500 error bij het inloggen na session timeout

Stap 7: Maak een docs/refs/ map

mkdir -p docs/refs

Begin met lege bestanden per categorie. Vul ze aan zodra Claude een fout maakt -- elke regel komt uit een echte les.

Belangrijk

Kopieer deze voorbeelden niet klakkeloos. De kracht van docs/refs/ zit in regels die uit jouw projectfouten komen. Een lege tabel met de juiste categorieën is waardevoller dan tien gekopieerde regels.

Stap 8: Stel .gitignore in

# Voeg toe aan .gitignore
.env
.env.local
.claude/worktrees/
node_modules/

.env.example -- veilig secrets delen

Ship een .env.example zonder waarden mee in git. Nieuwe teamleden kopieren dit naar .env en vullen hun eigen keys in.

# .env.example -- check dit WEL in naar git
DATABASE_URL=
API_KEY=
STRIPE_SECRET_KEY=
Secret gelekt? Doe dit onmiddellijk
  1. Draai de key in bij de provider (Stripe, AWS, GitHub, etc.) -- dit is stap 1, niet stap 3.
  2. Verwijder uit git history:
    git filter-repo --invert-paths --path <bestand-met-secret>
  3. Force push na overleg met je team:
    git push --force --all
  4. Voeg het bestand toe aan .gitignore zodat het niet opnieuw kan gebeuren.

Een secret dat ooit in git heeft gestaan is gecompromitteerd, ook als je het daarna verwijdert. Bots scannen publieke repo's continu op gelekte keys.

Verder groeien

Na deze 8 stappen heb je een solide basis. De volgende niveaus:

  • Agents: Bouw een gespecialiseerde agent voor terugkerende controles -- denk aan link-checks, SEO-audits, of content freshness.
  • Feedback loop: Fout → regel in docs/refs/. Herhaalde taak → skill. Terugkerende controle → agent. Snoei je CLAUDE.md regelmatig en experimenteer met modellen en parallel werk.

De volledige project-setup checklist

Dit is de inhoud van ~/.claude/docs/project-setup.md -- het document waar je globale CLAUDE.md naar verwijst (SSOT). Doorloop deze stappen bij een nieuw project of wanneer een project geen CLAUDE.md heeft.

SSOT

Dit document is de autoritieve bron voor project-inrichting. De globale CLAUDE.md verwijst ernaar, kopieert het niet. Wijzigingen hoef je maar op een plek te doen.

Checklist stap 1: Project CLAUDE.md aanmaken

Stel /init voor als er geen CLAUDE.md in de projectroot staat. De CLAUDE.md moet bevatten (compact, max ~100-150 regels):

ProjectWat doet het project, voor wieVerplicht
Tech stackFramework, taal, database, hostingVerplicht
Commando'sbuild, dev, test, lint -- exact zoals ze gedraaid wordenVerplicht
ArchitectuurMappenstructuur, belangrijke patronen, routingAanbevolen
Kritieke regelsDingen die NOOIT mogen (bijv. "verwijder geen content", "geen secrets")Verplicht
ReferentiesVerwijzingen naar docs/refs/ bestanden voor detailsAanbevolen

Wat er NIET in hoort: code style (dat doet de linter), code voorbeelden (verouderen), info die uit de code afgeleid kan worden.

Checklist stap 2: .gitignore controleren

Check of .gitignore aanwezig is met minimaal:

.env
.env.*
node_modules/          # of equivalent (venv/, vendor/)
dist/                  # build output (.astro/, .next/, build/)
.DS_Store
.vscode/settings.json
.idea/

Waarschuw als .gitignore ontbreekt of .env niet erin staat.

Checklist stap 3: Git initialiseren

Als er nog geen .git/ is:

git init
git checkout -b feature/<beschrijvend>
# Werk NOOIT direct op main

Checklist stap 4: docs/refs/ aanmaken

Maak docs/refs/ directory aan (leeg is OK bij start). Dit is de plek voor "littekens" -- lessen uit fouten:

BestandWanneer aanmaken

Checklist stap 5: Hooks instellen

Detecteer de tech stack en stel de juiste pre-commit hook voor:

StackPre-commit hook commando
Astronpx astro build
Next.js / Reactnpm run lint && npx tsc --noEmit
Pythonruff check . && mypy .
Gogo vet ./... && staticcheck ./...
Flutterflutter analyze && dart format --set-exit-if-changed .
Generieknpm run build of npm test

Installeer als git hook:

mkdir -p .git/hooks
cat > .git/hooks/pre-commit << 'HOOK'
#!/bin/sh
echo "Running pre-commit checks..."
# COMMANDO HIER (zie tabel hierboven)
if [ $? -ne 0 ]; then
  echo "Pre-commit check failed! Commit blocked."
  exit 1
fi
HOOK
chmod +x .git/hooks/pre-commit

Checklist stap 6: Eerste commit

Na het inrichten: selectief stagen (specifieke bestanden, niet git add .), duidelijk commit bericht, verifieer dat de pre-commit hook werkt.

Automatische check: SessionStart hook

De globale settings (~/.claude/settings.json) bevat een SessionStart hook die bij elke sessie automatisch checkt of CLAUDE.md, docs/refs/, .gitignore en .git aanwezig zijn. Dit is deterministisch -- het draait altijd. Zie Hooks -- SessionStart voor de configuratie en het script.

Verificatie

Na het doorlopen van alle stappen, controleer:

  • CLAUDE.md aanwezig en compact
  • .gitignore met .env en secrets
  • Git repo met feature branch (niet main)
  • docs/refs/ directory aanwezig
  • Pre-commit hook geinstalleerd en werkend
  • SessionStart hook actief (check via /hooks)
  • Eerste commit geslaagd

Eerste sessie checklist

Geef dit aan iedereen die voor het eerst met Claude Code werkt:

  1. Open je terminal in het project: cd mijn-project && claude
  2. Verken het project: "Geef me een overzicht van dit project"
  3. Start een feature: /start-feature [beschrijving]
  4. Claude maakt een branch, plan, en wacht op goedkeuring
  5. Review het plan -- stel vragen, vraag aanpassingen
  6. Zeg "ga door" -- Claude implementeert, test, en commit
  7. Check de draft PR op GitHub
  8. Wissel van taak? Typ /clear eerst
  9. Bug gevonden? /fix-bug [beschrijving met error]
  10. Sessie langer dan 30 min? Typ /compact
Check dit in naar git

De .claude/ map, CLAUDE.md en docs/refs/ horen in version control. Zo werkt iedereen die het project cloned meteen volgens dezelfde werkwijze -- zonder iets te hoeven configureren.
